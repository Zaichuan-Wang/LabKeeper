# 数据质量与维护能力改造审核稿

本文档整理下一轮最值得优先补的上线前质量与维护能力。目标是防止真实库存数据变脏、保证可回退、提升追溯可读性，而不是引入复杂审批流或重型 LIMS 规则。

当前建议优先做：

1. 编号 + 管号重复检查。
2. 管理员数据健康检查报告。
3. 数据库备份策略。
4. 实物详情时间线。

## 总体原则

- 不改变当前轻量架构：SQLite 单库、Python 后端、普通 HTML/CSS/JS 前端。
- 第一版以“提醒、阻断明显错误、给管理员可读报告”为主。
- 不自动修复历史数据，避免误改正式库。
- 对日常录入人员保持直观提示，避免把界面变成审计系统。
- 对批量导入、迁移、上线更新这类高风险操作，优先保证可回退。
- 日常试剂/标本 Excel 导入、编辑、移动和出库已经放在“库存空间 → 批量处理”，并已有“读取 Excel → 预检查 → 确认提交”的交互，本计划不再重复设计管理员 Excel 导入预检。
- `dev_tools/` 只用于部署前临时测试、数据导入或一次性离线维护；正式部署后，系统维护不能依赖 `dev_tools/` 里的脚本。
- 需要部署后持续使用的能力，必须进入后端正式模块、管理员界面或部署文档中的系统级计划任务。
- 不为了这些功能重写现有库存、移动、出库、验证、到货的核心业务流程。

## 1. 编号 + 管号重复检查

### 背景

`clinical_samples.code` 允许重复是合理的，因为同一个临床标本可以分成多管。真正应该唯一的是：

```text
code + aliquot_no
```

如果同一标本号下出现两个相同管号，后续移动、出库、盘点和扫码都会变得含糊。

### 修改目标

- 新建标本、分装、编辑标本时，禁止产生重复的 `code + aliquot_no`。
- 如果历史库中已经存在重复，先在数据健康报告里提示，不在迁移时自动删除或合并。
- 用户手动编辑标本号时，如果会和已有管号冲突，应直接拒绝保存。

### 后端建议

涉及文件：

- `backend/clinical_samples.py`
- `backend/database.py`

建议新增检查函数：

```python
def ensure_unique_sample_aliquot(conn, code, aliquot_no, exclude_id=None):
    ...
```

检查逻辑：

- `code` 为空时直接报错。
- `aliquot_no` 为空时按当前逻辑通常不应出现；如果历史数据为空，健康检查提示。
- 查询同一 `code`、同一 `aliquot_no`、不同 `id` 的记录。
- 如果存在，返回 `409`：

```text
该标本号下已存在相同管号，请修改标本号或管号。
```

数据库层可选增强：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_clinical_samples_code_aliquot
ON clinical_samples(code, aliquot_no)
WHERE aliquot_no IS NOT NULL;
```

注意：如果当前正式库已有重复，不能直接创建唯一索引。应先通过健康检查列出重复，人工处理后再加索引。

### 前端建议

第一版不用新增复杂界面。保存失败时展示后端错误即可。

后续可在编辑标本号时增加轻提示：

```text
修改标本号会影响同一标本下的分装关系和统计，请确认。
```

### 验证方式

- 新建同一 `code` 下多管，应生成连续不重复管号。
- 手动编辑某管到已存在的 `code + aliquot_no`，应失败。
- 分装时生成多管，不应跳出重复管号。
- 历史库有重复时，健康检查能列出重复组。

## 2. 管理员数据健康检查报告

### 背景

正式运行后，数据问题通常不是一次爆炸，而是慢慢积累：位置残留、孔位重复、状态和位置矛盾、导入数据不规范等。管理员需要一个“只读体检页”，先看到问题，再决定是否人工修复。

### 修改目标

新增管理员只读接口：

```text
GET /api/admin/data-health
```

管理员页新增“数据健康”标签，显示每类问题数量和前若干条示例。

### 第一版检查项

建议先做以下 7类：

1. 已耗尽试剂仍占位置。
2. 已耗尽标本仍占位置。
3. 已订购试剂仍占位置。
4. 可用或停用库存指向不存在的空间。
5. 同一空间同一孔位被多个库存对象占用。
6. `code + aliquot_no` 重复。
7. 同一货号对应多个明显不同名称。

### 返回结构草案

```json
{
  "items": [
    {
      "key": "consumed_reagent_has_location",
      "label": "已耗尽试剂仍占位置",
      "severity": "error",
      "count": 2,
      "examples": [
        {
          "id": 12,
          "code": "RG000012",
          "name": "Anti-GAPDH",
          "location": "-80冰箱 / 盒A1；B3",
          "message": "状态为已耗尽，但仍有 storage_node_id 或 position_in_box"
        }
      ]
    }
  ],
  "summary": {
    "errors": 3,
    "warnings": 5,
    "checked_at": "2026-06-12 14:30:00"
  }
}
```

### 后端建议

新增文件：

- `backend/data_health.py`

在 `backend/server.py` 的管理员路由中增加：

```text
GET /api/admin/data-health
```

该接口只读，不写数据库，不自动修复。

### 前端建议

涉及文件：

- `frontend/index.html`
- `frontend/admin-page.js`
- `frontend/styles.css`

管理员页面增加一个标签：

```text
Excel | 用户 | 选项 | 数据健康
```

展示方式：

- 顶部显示：错误数、警告数、检查时间。
- 每类问题一个折叠区或小表格。
- 每类最多展示前 20 条，避免页面太长。
- 不提供“自动修复”按钮。

### 验证方式

- 正常库返回 0 个 error 时页面应清楚显示“未发现明显问题”。
- 构造重复孔位、重复管号、已耗尽仍占位等情况，接口能列出示例。
- 普通用户访问应返回 403。

## 3. 数据库备份策略

### 背景

数据库备份要解决三个问题：

- 运行中也能生成一致快照，避免 SQLite WAL 模式下只复制主库文件导致备份不完整。
- 批量导入、部署更新、迁移前必须有可回退点。
- 非计算机专业维护者能通过明确入口完成备份和下载，而不是依赖命令行脚本。

### 基本原则

- 使用 SQLite backup API 生成一致快照。
- 不把 `db/lab_inventory.sqlite3`、`db/lab_inventory.sqlite3-wal`、`db/lab_inventory.sqlite3-shm` 简单复制当作正式备份策略。
- 备份文件放入 `db/backups/` 或服务器指定备份目录。
- 每次备份后运行 `PRAGMA integrity_check`。
- 恢复操作第一版不做网页一键恢复，避免误覆盖正式库。
- 恢复必须人工确认、停服务、替换库、删除旧 WAL/SHM、重启并检查。

### 后端建议

新增正式后端模块：

```text
backend/backup.py
```

它不是临时脚本，而是部署后可持续使用的正式后端能力。

建议提供函数：

```python
def create_database_backup(reason: str, user: dict | None = None) -> dict:
    ...

def list_database_backups(limit: int = 50) -> dict:
    ...

def verify_database_backup(path: str) -> dict:
    ...
```

备份返回结构示例：

```json
{
  "path": "db/backups/lab_inventory_before_excel_import_20260612_143000.sqlite3",
  "filename": "lab_inventory_before_excel_import_20260612_143000.sqlite3",
  "reason": "before_excel_import",
  "size": 1234567,
  "created_at": "2026-06-12 14:30:00",
  "integrity_check": "ok",
  "table_counts": {
    "reagents": 120,
    "clinical_samples": 2783,
    "storage_nodes": 42,
    "movements": 260
  }
}
```

### 管理员接口建议

建议新增管理员接口：

```text
GET  /api/admin/backups
POST /api/admin/backups
GET  /api/admin/backups/{filename}/download
```

第一版能力：

- 查看最近备份列表。
- 手动创建备份。
- 下载某个备份文件。

不做：

- 网页一键恢复。
- 自动删除正式库。
- 直接上传备份并覆盖当前库。

### 触发场景

自动触发：

- 管理员页的“数据库表导入”属于迁移、备份恢复或工程修复入口。如果继续保留该高危写入口，正式写入前应自动调用 `create_database_backup()`。

手动触发：

- 管理员页面新增“数据库备份”区。
- 提供“立即备份”按钮。
- 显示最近备份列表。
- 显示备份原因、时间、大小、完整性检查结果。
- 提供下载按钮。

定时触发：

- 正式服务器建议每日定时备份(可调整间隔)。
- 由于 `dev_tools/` 不能作为部署后维护依赖，定时备份应走正式后端接口或部署文档明确的正式运维命令。
- 如果采用服务器计划任务调用 HTTP 接口，应使用受控管理员令牌或本机维护令牌。

### 保留策略建议

保留时间应可以手动调整

### 恢复流程建议

恢复不通过网页按钮完成。部署文档中写清楚人工恢复流程：

1. 通知用户暂停使用系统。
2. 停止后端服务。
3. 先把当前库再做一次应急备份。
4. 替换 `db/lab_inventory.sqlite3`。
5. 删除同目录旧的 `lab_inventory.sqlite3-wal` 和 `lab_inventory.sqlite3-shm`。
6. 启动后端。
7. 运行 `backend/server.py --check`。
8. 打开页面抽查首页、空间概览、明细查询、流转记录。

### 涉及文件

预计涉及：

- `backend/backup.py`：正式备份能力。
- `backend/admin.py`：高危数据库表导入如继续保留，写入前调用备份；或导出备份列表辅助函数。
- `backend/server.py`：管理员备份接口。
- `frontend/admin-page.js`：管理员备份区。
- `frontend/index.html`：管理员备份标签或面板。
- `frontend/styles.css`：备份列表样式。
- `SERVER_MANUAL_UPDATE.md`：恢复流程和定时备份说明。
- `USER_MANUAL.md`：管理员备份入口说明。

### 验证方式

- 管理员能在页面手动创建备份。
- 备份使用 SQLite backup API 生成，运行中也能得到一致快照。
- 备份后返回 `integrity_check = ok`。
- 普通用户不能访问备份接口。
- 部署文档明确恢复步骤，且不要求运行 `dev_tools/`。

## 4. 实物详情时间线

### 背景

当前详情页主要展示对象的当前状态。时间线目标是让用户打开任意一个实物时，能看到它从创建到现在发生过的关键事件：

- 何时创建或入库。
- 放到哪里。
- 是否移动过。
- 是否出库或耗尽。
- 是否回滚过。
- 谁操作。
- 试剂是否有关联到货和货号级验证记录。

第一版定位为“只读聚合视图”，不新增流程，不改变历史写入逻辑。

### 当前差异

试剂详情现状：

- 当前信息。
- 验证记录。
- 到货记录。
- 移动记录。

但这些记录是分散表格，不是统一时间顺序。

标本详情现状：

- 标本号。
- 管号。
- 类型。
- 状态。
- 重量/体积。
- 入库日期。
- 位置。
- 编辑、移动、出库按钮。

它没有统一展示移动、出库、回滚和编辑历史。

### 后端建议

新增正式后端模块：

```text
backend/inventory_timeline.py
```

建议接口：

```text
GET /api/inventory/timeline?item_type=reagent&id=123
GET /api/inventory/timeline?item_type=sample&id=456
```

该接口只读，不写数据库。

返回结构草案：

```json
{
  "item": {
    "item_type": "sample",
    "id": 456,
    "code": "SP000001",
    "title": "SP000001 · 灌洗液"
  },
  "items": [
    {
      "time": "2026-06-12 10:30:00",
      "event_type": "movement",
      "title": "位置移动",
      "summary": "从 未归位 移动到 -80冰箱 / 盒A；B3",
      "actor": "张三",
      "from_location": "未归位",
      "to_location": "-80冰箱 / 盒A；B3",
      "related_table": "movements",
      "related_id": 88
    }
  ],
  "count": 1
}
```

### 试剂时间线事件来源

试剂时间线建议聚合：

1. `reagents` 自身创建记录。
2. `orders` 订购来源，如果能通过 `arrivals.order_id` 关联。
3. `arrivals` 到货入库记录。
4. `validations` 货号级验证记录。
5. `movements` 移动、出库、回滚记录。
6. `audit_logs` 信息编辑记录。

### 标本时间线事件来源

标本时间线建议聚合：

1. `clinical_samples` 自身创建或入库记录。
2. 同一 `code` 下的分装关系提示。
3. `movements` 移动、出库、回滚记录。
4. `audit_logs` 信息编辑记录。

### 货号级验证的表达

试剂验证继续以货号级验证为主。时间线中应明确显示：

```text
货号验证
```

而不是写成“本管验证”。

推荐文案：

```text
货号验证：CST 5174 在 WB 中结果为通过。
```

原因：

- 抗体、试剂盒、引物等验证通常是产品/货号级判断。
- 同一货号后续重复采购可以共享历史验证参考。
- 不应强行要求每一支管单独验证。

未来如果确实需要批次级验证，可以新增可选字段 `lot_no` 或可选关联 `reagent_id`，但不替代 `catalog_no` 主逻辑。

### 前端建议

在现有详情弹窗中增加“时间线”区，同时移除不必要的记录显示，不新增大页面。

展示结构：

```text
当前信息卡片
操作按钮

时间线
2026-06-12  出库登记：从 -80冰箱 / 盒A；B3 出库，原因：实验使用
2026-06-10  位置移动：从 未归位 移动到 -80冰箱 / 盒A；B3
2026-06-09  标本入库：登记 3 管，第 1 管
```

建议在 `frontend/inventory-page.js` 中新增：

```javascript
async function loadInventoryTimeline(itemType, id) { ... }
function renderInventoryTimeline(events) { ... }
```

第一版可以在打开详情弹窗时同步请求时间线。若后续性能压力明显，再改成懒加载。

### 审计记录展示边界

`audit_logs` 可能包含 JSON 旧值和新值。第一版不展开所有字段差异，只显示：

```text
信息修改：管理员更新了该对象信息。
```

如果能安全识别字段，可展示少数关键字段：

- 状态。
- 数量/体积。
- 存放位置。
- 备注。

不建议第一版做复杂 JSON diff。

### 不改数据库

第一版不新增表，不改现有业务写入。

不做：

- 独立事件表。
- 通用事件状态机。
- 时间线事件回滚。
- 自动补全历史缺失信息。
- 把验证从货号级强制改成试剂管级。

### 涉及文件

预计涉及：

- `backend/inventory_timeline.py`：统一时间线聚合。
- `backend/server.py`：新增只读时间线路由。
- `frontend/inventory-page.js`：详情弹窗请求和渲染时间线。
- `frontend/styles.css`：时间线样式。
- `USER_MANUAL.md`：说明详情页时间线含义。

### 验证方式

- 打开试剂详情能看到当前信息和按时间排序的历史事件。
- 打开标本详情能看到当前信息和移动、出库、回滚等事件。
- 验证记录在时间线中标明为“货号验证”。
- 已出库或已耗尽对象仍能查看历史。
- 没有历史事件时显示“暂无历史记录”，不报错。
- 普通有查看权限的用户能查看时间线；没有明细权限的用户不能通过接口绕过限制。

## 建议实施顺序

推荐顺序：

1. 先做管理员数据健康检查报告。
2. 再做标本号 + 管号重复检查。
3. 再做实物详情时间线。
4. 最后做数据库手动备份和备份列表；如果管理员高危数据库表导入继续保留，再接入写入前备份。

原因：

- 健康检查是只读功能，风险最低，也能先摸清当前库有没有历史问题。
- 如果当前库已经有重复管号，先展示出来，再决定是否清理和加唯一索引。
- 时间线也是只读聚合，用户价值明显，风险较低。
- 备份模块要注意路径权限、下载权限和恢复说明，适合在只读功能之后做。
- 日常 Excel 库存导入已经由“批量处理”的预检查流程覆盖，不再作为独立改造项；管理员数据库表导入只保留工程维护定位，不扩成日常业务入口。

## 不做事项

第一版不做：

- 自动修复数据。
- 审批流。
- 复杂角色权限扩展。
- 空间级授权。
- 全量数据质量评分。
- 重做日常 Excel 批量导入预检流程。
- 把管理员高危数据库表导入扩成日常业务导入入口。
- 导入后自动覆盖孔位。
- 对历史数据做静默合并。
- 网页一键恢复数据库。
- 把验证从货号级强制改成试剂管级。
- 为时间线新增独立事件表或通用状态机。

## 最小验收标准

- 普通用户不会感知额外复杂度。
- 管理员能看到当前库是否存在明显数据风险。
- 标本号 + 管号不会继续产生新重复。
- 试剂和标本详情能看到只读时间线。
- 管理员可以通过正式后端能力创建一致数据库备份。
- 日常库存 Excel 批量处理保持先预检查、再确认提交；管理员高危数据库表导入不作为普通维护入口。
- 如果保留管理员高危数据库表导入，必须在写入前自动备份，或在界面和部署文档中明确要求先使用正式备份入口创建一致快照。
- 所有失败提示能让非计算机专业维护者看懂下一步该怎么处理。
